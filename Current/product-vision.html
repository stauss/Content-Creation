Vision Overview

Current is a source-connection and synthesis platform. It is not primarily a bookmarking tool, not a read-later system, and not a retrieval layer over saved links. The core user action is simple: connect a source to a Stream. From that point forward, the system monitors, classifies, scores, clusters, and synthesizes incoming content into Digests and Podcasts.

This product is intentionally built in six phases — starting with a focused connection and ingestion system and expanding into a full multi-surface editorial intelligence platform. The early phases prove the core value loop. The later phases make it durable and broadly accessible.

The goal is not to build another read-later app. The goal is to create an editorial intelligence layer that sits between the raw flood of information and a user's actual understanding of what's happening — one that attributes every insight back to the sources that earned it.

The Core Loop
Connect — Link sources to a user-defined Stream
Monitor — System evaluates and classifies continuously
Synthesize — AI generates Digests and editorial Podcasts
Attribute — Every insight traces back to its sources

The Problem
Information Overload
Users collect sources but never synthesize them
Read-later tools and RSS readers accumulate content without producing meaning. Users save links to piles they never return to. Important information gets lost not because it wasn't saved — but because there was never a system to surface what actually mattered from inside the pile.
Context Collapse
Stories emerge from patterns, not single articles
The most important developments in any domain rarely appear in one place. They emerge from clusters of reporting, commentary, and adjacent data — spread across dozens of sources. No existing tool helps users understand a developing story as a whole. They're left doing the synthesis themselves, manually, with no infrastructure to support it.
The Trust Gap
AI summaries are confident but ungrounded
Existing AI tools generate fluent, confident summaries — but make it nearly impossible to verify where claims came from. Users can't trust what they can't inspect. Source attribution is treated as an afterthought, not an architectural requirement. When something in a summary is wrong or overstated, there's no path back to reality.
Conceptual Model
User-Facing
Drop
A connected source item or captured content unit. The atomic input of the system — a URL, an RSS item, or an uploaded document connected to a Stream.
User-Facing
Stream
A user-defined intent bucket. Drops flow into Streams. Each Stream has its own cadence, thresholds, and editorial profile. The user's primary organizing surface.
User-Facing
Digest
A scheduled or AI-triggered synthesis output. A text summary with source references — and the planning layer from which Podcasts are generated.
User-Facing
Podcast
A generated narrative audio output. Built from weighted thematic segments — not a flat read-back of recent items. Each segment maps to sources the user connected.
System-Facing
Branch
A hidden thematic subdivision inside a Stream. The AI's working editorial layer — used to cluster related Drops and plan segment allocation in Podcasts.
System-Facing
Rogue Wave
An unscheduled Digest triggered when content significance spikes beyond threshold. The system's mechanism for surfacing major developments without waiting for the next cycle.
The Core Insight

Users don't need another place to save content. They need a system that does something with it — one that synthesizes intelligently, attributes honestly, and doesn't require them to do the editorial work themselves.

The instinct with information tools is to start with retrieval: save everything, surface later, read when ready. Current inverts this. Synthesis is the primary output. The Podcast and the Digest are not features bolted onto a feed — they are the product. Sources are the grounding that makes them trustworthy.

This introduces a new category that sits between two inadequate models: the read-later pile that never gets read, and the AI summary that can't be traced back to anything real.

The Design Principle

Users direct the Stream. The system shapes the current.

Two Inadequate Models
The read-later pile — saved, never read, never synthesized
The AI summary — confident, fluent, and unverifiable
The New Category
Synthesis grounded in sources the user controls
Every insight attributable — traceable to its origin
Product Phases
Foundation Phase 01 — Connect

Establish the TypeScript monorepo, Convex backend, shared packages, and app shells. This phase creates the infrastructure that all future phases build on. Nothing meaningful ships without this foundation being solid — shared domain types, zod schemas, and design tokens must be locked before feature endpoints proliferate.

Packages & Deliverables
Turborepo + pnpm monorepo Convex schema + auth Shared domain package Shared zod schema package Shared UI primitives Design token package Stream + Drop CRUD Web app shell Chrome extension shell
Architecture Principles
Shared First
Domain logic, types, and tokens live in packages — not in apps.
Lock Early
Domain vocabulary and zod schemas must be stable before features grow.
Typed Throughout
TypeScript strict mode from day one. No escape hatches at boundaries.
Shell First
Web + extension shells must be working end-to-end before features begin.
Key Risks
Monorepo discipline — shared package boundaries must be enforced early; boundary debt compounds fast
Schema drift — domain types and zod schemas must co-evolve; divergence becomes painful quickly
Connection System Phase 02 — Capture

Build the end-to-end connection workflow. The Chrome extension becomes the fastest capture surface — users connect the current page to a Stream in one or two actions. The ingestion pipeline extracts, normalizes, and deduplicates content so every Drop becomes usable source material. This phase proves the core input loop before any synthesis is attempted.

Features & Components
Extension connect flow Stream suggestion logic Inline Stream creation URL ingestion pipeline Content extraction + normalization Drop deduplication Source trust baseline Feed monitoring (RSS)
Extension UX Principles
Page-Aware
Opens knowing the current page. No manual input required to begin.
Confident
Suggests a Stream with a reason. Confidence over ambiguity.
Fast
Connect in two taps. The extension is not a management surface.
Focused
No podcast playback, no digest review. Capture only.
Key Risks
Extraction quality — content pulled from URLs varies wildly; extraction must be robust to real-world site complexity
Stream suggestion accuracy — a low-confidence suggestion degrades trust in the entire extension experience
Digest Engine Phase 03 — Synthesize

Add the scheduling system and the first synthesis output. Users receive scheduled Digests that summarize what has happened across their Streams — grounded in the sources they connected, not AI fabrication. This phase validates the core synthesis value proposition in its simplest form before introducing the full editorial podcast layer.

Features & Components
Convex scheduling system Strict schedule mode Summary generation (AI) Source references in Digest Digest list view Digest detail view Read / unread state Unified search (Streams + Drops + Digests)
Key Risks
Grounding discipline — digest summaries must only claim what sources support; hallucinated claims erode trust permanently
Cadence calibration — too-frequent digests feel noisy; too-infrequent feels stale; defaults must be tuned carefully
Source attribution in text — references must be surfaced clearly in the Digest UI, not buried or omitted
Versioning AI outputs — prompt versions and model versions must be stored with every generation from the start
Podcast Intelligence Phase 04 — Editorial

Transform Digests into editorial, segment-driven podcasts. This is the phase where Current becomes editorially distinct. Podcasts are not a flat read-back of recent Drops — they are built from weighted thematic Branches, editorial synthesis, and source overlap handling. Playtime is allocated by importance. Sources map to segments in a many-to-many relationship. Every claim in a podcast segment is tied to the Drops that informed it.

Features & Components
Branch assignment (AI) Drop scoring engine Segment planner (AI) Podcast script generation Audio generation Segment timeline UI Source attribution layer Follow-up Digest chat Scoring package (versioned)
Scoring Model Direction
Drop score = (relevance × 0.30) + (impact × 0.20) + (novelty × 0.18) + (recency × 0.12) + (trust × 0.10) + (cross-branch × 0.10) − (redundancy × 0.20)
Branch score = sum(top drop scores) × branch momentum × stream alignment — feeds directly into segment playtime allocation
Critical Rules
Source-to-segment must be many-to-many — a single Drop can anchor one segment, support another, and provide context in a third
Podcast must never feel like a news ticker — editorial synthesis and "why this matters" framing are non-negotiable quality requirements
Adaptive Intelligence Phase 05 — Reactive

Layer in event-driven generation. The system learns to react when content significance spikes — triggering Overflow or Rogue Wave Digests without waiting for the next scheduled cycle. Users gain meaningful control over sensitivity through threshold settings in the web app and directly inside the extension popup. This phase transforms Current from a scheduled tool into a genuinely responsive intelligence layer.

Features & Components
Overflow threshold triggers Rogue Wave detection Adaptive Flow mode Threshold controls (global) Stream-level threshold overrides Extension threshold editing Immediate notification support
Key Risks
Alert fatigue — Rogue Wave thresholds that are too sensitive create noise; defaults must err toward restraint
Threshold UX complexity — controls must feel meaningful, not technical; abstraction over raw numbers is essential
Multi-Surface Expansion Phase 06 — Scale

Expand Current to mobile and desktop native. The mobile experience centers on consumption: listening to podcasts, reading Digests, inspecting segment sources, and asking follow-up questions. The desktop native shell adds media controls, tray integration, offline caching, and faster access. Across all surfaces, the product should feel like one thing — optimized for the job each surface does best.

Platforms & Features
Mobile app (Expo / React Native) Digest feed + detail (mobile) Podcast player (mobile) Segment inspector sheet Follow-up chat (mobile) Desktop native (Tauri shell) OS notifications Media key integration Offline digest caching
Surface Design Rules
Mobile optimizes for listening, reading, and lightweight connection — not for Stream management or advanced controls
Desktop native reuses the web client where practical (Tauri) — a fully specialized desktop client is a later consideration if needed

Defining
V1 Scope — Phases 1–3
Must Have
The Non-Negotiable Loop
Monorepo and Convex schema. Stream creation and Drop connection. Chrome extension with suggested Stream logic. URL ingestion and content extraction. Strict-schedule Digest generation grounded in connected sources. Web app shell with stream list, digest list, and digest detail views. Without these, nothing else is meaningful to build.
Nice to Have
Add Once the Loop Is Solid
Basic podcast generation and segment timeline UI. Branch visualization inside the web app. Threshold controls in the extension popup. Follow-up Digest chat. These deepen the product meaningfully — but only once the core connection and synthesis loop is proven and friction-free.
Out of Scope for V1
Deliberately Deferred
Mobile app and desktop native packaging (Phase 6). Adaptive Flow and Rogue Wave triggers (Phase 5). Team workspaces, shared Streams, and social features. V1 is a personal tool. Networked and multi-user features add complexity before the core value loop is proven — and dilute the focus that makes it good.

Final Thought

This product works because it starts with a real problem — not with what users wish they did with their reading list, or how they imagine a perfect information diet might function in theory, but with how information actually arrives in practice: in floods, in fragments, and almost always without any system to turn it into coherent understanding.

Every phase of Current is built to close that gap. Phase 1 and 2 make the connection and ingestion loop fast and trustworthy. Phase 3 turns that input into a grounded synthesis output. Phases 4 and 5 add the editorial intelligence and reactivity that makes it genuinely useful over time. Phase 6 brings that experience to every surface a user actually lives on.

Build Current like a durable product system, not a demo. The product should ultimately feel calm, intelligent, and beautifully structured — while the underlying architecture remains rigorous enough to support years of growth.
